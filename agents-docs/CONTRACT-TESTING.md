# Contract Testing Guide

Contract testing ensures that a change in one service (the Provider) doesn't break another service that depends on it (the Consumer). This is especially critical in AI-driven development where interfaces might be refactored autonomously.

## What is Pact?

[Pact](https://pact.io) is a code-first tool for contract testing HTTP and message integrations. It is "consumer-driven," meaning the consumer defines the contract by writing a test.

## Why Broker-Free?

This repository uses Pact in **broker-free mode**. Instead of a central Pact Broker server, pact files are:
1. Generated by consumer tests.
2. Committed to `contracts/pacts/` in this repository.
3. Versioned with the code in Git.

## Multi-Language Support

Pact's core is written in Rust (`pact_ffi`) and exposes a Foreign Function Interface (FFI) that allows any language to use it. Supported languages include Rust, TypeScript, C#, Python, C++, Go, and more.

## Writing Your First Pact Consumer Test

See the `templates/` directory for starter files:
- `templates/contract-test-rust.rs`
- `templates/contract-test-typescript.ts`
- `templates/contract-test-csharp.cs`
- `templates/contract-test-cpp.cpp`

### General Flow
1. **Define the interaction**: Set up a mock server with the expected request and response.
2. **Execute the request**: Run your client code against the mock server.
3. **Verify**: Assert that the mock server received the expected request.
4. **Write pact file**: Save the recorded interaction to `contracts/pacts/`.

## Verifying a Provider

Provider verification ensures the actual implementation matches the contract.

1. Start your provider service (e.g., `npm start` or `cargo run`).
2. Run the verification script:
   ```bash
   bash tests/verify-pact-contracts.sh
   ```

## CI Integration

Contract validation and provider verification are automated via `.github/workflows/contract-testing.yml`.

## Breaking Changes: Deprecation Workflow

1. **Identify**: Use `@pact-contract-testing` skill to check if a change is breaking.
2. **Deprecate**: Add the new field/endpoint while keeping the old one.
3. **Update Consumers**: Update all consumer tests to use the new interface and generate new pacts.
4. **Remove**: Once all consumers have migrated, remove the old field/endpoint.
