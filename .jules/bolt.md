## 2026-05-24 - [perf: eliminate subshells in validation loops] Learning: [Replacing `find | sort` with native globbing (`for file in *.md`), `while read` appending with `mapfile`, and external formatting pipelines (`xargs`/`tr`) with native string manipulation (`${var//[^[:alnum:]]/_}`) significantly reduces process forking overhead. Also, when evaluating array length conditionally, `[[ ${#ARRAY[@]} -eq 0 ]]` is safer and more feature-rich than `[ ${#ARRAY[@]} -eq 0 ]`.] Action: [In high-frequency Bash validation loops, avoid subshells and process substitutions (`< <()`, `<<<`) when native loops and expansions are sufficient. Ensure `[[ ]]` is consistently used over `[ ]` for conditional tests for robust execution.]
## 2026-05-24 - [perf: eliminate string parsing subshells] Learning: [Replacing `IFS=":" read <<<` within loops with native parameter expansion (`${var%%:*}`), and pre-parsing delimiter-separated strings into bash arrays, drastically reduces parsing overhead. However, using native lowercase expansion (`${var,,}`) breaks compatibility with Bash 3.2 (macOS default), so `tr` must be maintained for cross-platform portability.] Action: [In tight bash validation loops, eliminate `read <<<` parsing in favor of `%%` and `#` expansions or pre-parsed arrays, but strictly avoid Bash 4.0+ string modification syntax to ensure 3.2 compatibility.]
## 2026-05-28 - [perf: optimize command invalidation rule filtering] Learning: [When evaluating a set of items against a list of rules in a multi-nested validation loop, pre-filtering the rules based on the primary condition (e.g., file matching) before iterating over the items significantly reduces the number of operations from O(N*M) to O(N + K*M) where K is the small number of matching rules.] Action: [In nested validation loops, always pre-filter the outer loop conditions if they are independent of the inner loop items to avoid redundant inner loop evaluations.]
